> 规则模式（Rule）运行，节点组见下文。

## 一、最常见故障：节点选择组选到了死节点

### 症状
- GitHub / 新终端的 Claude 连不上
- 老终端的 Claude 还能用（TCP 连接已建立，不受影响）
- `curl -x http://127.0.0.1:7891 https://www.google.com` 失败

### 原因
流量走向：
```
GitHub / 新连接 → 漏网之鱼 → 🚀 节点选择 → [当前节点]（若挂了就失败）
Claude（已有会话）→ anthropic 专属规则 → ♻️ 自动选择 → 香港节点（可能仍存活）
```
老终端 Claude 能用，是因为 TCP 连接早就建立了，不需要重新走代理节点。

### 一键修复：切到自动选择
```bash
curl -s -X PUT http://127.0.0.1:9091/proxies/%F0%9F%9A%80%20%E8%8A%82%E7%82%B9%E9%80%89%E6%8B%A9 \
  -H "Content-Type: application/json" \
  -d '{"name":"♻️ 自动选择"}'
```

### 验证是否恢复
```bash
curl -x http://127.0.0.1:7891 --max-time 10 https://www.google.com -o /dev/null -w "%{http_code}"
# 返回 200 即正常
```

## 二、快速诊断流程（AI 和自己都看这个）

```bash
# 步骤1：Clash 是否在跑？
ss -tlnp | grep 7891

# 步骤2：代理能通外网吗？
curl -x http://127.0.0.1:7891 --max-time 8 https://www.google.com -o /dev/null -w "%{http_code}"

# 步骤3：看当前节点选择组用的哪个节点
curl -s http://127.0.0.1:9091/proxies/%F0%9F%9A%80%20%E8%8A%82%E7%82%B9%E9%80%89%E6%8B%A9 | python3 -c "import json,sys; d=json.load(sys.stdin); print('当前:', d['now'])"

# 步骤4：列出所有可选节点
curl -s http://127.0.0.1:9091/proxies/%F0%9F%9A%80%20%E8%8A%82%E7%82%B9%E9%80%89%E6%8B%A9 | python3 -c "import json,sys; d=json.load(sys.stdin); [print(' ', n) for n in d['all']]"
```

**结果判断：**
| 步骤2结果 | 步骤3结果 | 结论 |
|-----------|-----------|------|
| 200 | 任意 | 代理正常，问题在别处 |
| 非200 | ♻️ 自动选择 | 自动选择的节点也挂了，换订阅或等待 |
| 非200 | 具体节点名 | 该节点挂了，执行一键修复切到自动选择 |
| 步骤1无输出 | - | Clash 没在跑，重启 Clash |

## 五、各工具代理配置

### ~/.bashrc（已配置，开终端自动生效）
```bash
proxy_on   # 手动开启
proxy_off  # 手动关闭
```

### git
```bash
git config --global http.proxy http://127.0.0.1:7891
git config --global https.proxy http://127.0.0.1:7891

# 取消 git 代理
git config --global --unset http.proxy
git config --global --unset https.proxy
```

### pip（临时）
```bash
pip install xxx --proxy http://127.0.0.1:7891
```

### npm
```bash
npm config set proxy http://127.0.0.1:7891
npm config set https-proxy http://127.0.0.1:7891
```

## 六、给其他用户共享代理（zhangjie 等）

在对方机器的 `~/.bashrc` 里加（把 `127.0.0.1` 换成本机局域网 IP）：

```bash
export http_proxy=http://<本机IP>:7891
export https_proxy=http://<本机IP>:7891
export HTTP_PROXY=http://<本机IP>:7891
export HTTPS_PROXY=http://<本机IP>:7891
export all_proxy=socks5://<本机IP>:7891
export no_proxy=localhost,127.0.0.1,::1,10.0.0.0/8,192.168.0.0/16
```

查本机 IP：`hostname -I | awk '{print $1}'`

## 七、订阅系统常见问题

### 症状：所有节点都挂了 / 订阅节点批量失效
使用订阅制服务时，机场会定期更换节点端口或 IP，若本地配置未同步则全部节点失效。

### 诊断：区分"节点挂了"和"订阅过期/失效"

```bash
# 看当前所有节点的延迟状态（0ms 或 timeout 说明节点不可用）
curl -s http://127.0.0.1:9091/proxies | python3 -c "
import json, sys
d = json.load(sys.stdin)
for name, info in d['proxies'].items():
    if info.get('type') not in ('Selector','URLTest','Fallback','LoadBalance','Direct','Reject','Compatible'):
        hist = info.get('history', [])
        delay = hist[-1]['delay'] if hist else 0
        print(f'{delay:5}ms  {name}')
" | sort -n | head -20
```

| 现象 | 原因 | 解决 |
|------|------|------|
| 少量节点 0ms | 部分节点挂掉 | 切换到自动选择 |
| 所有节点 0ms | 订阅节点批量更换端口/IP | **更新订阅** |
| 延迟正常但连不上 | DNS 污染 / 防火墙 | 换节点或换协议 |

### 更新订阅（重载配置）

```bash
# 方法一：通过 API 热重载配置（不中断已有连接）
curl -s -X PUT http://127.0.0.1:9091/configs \
  -H "Content-Type: application/json" \
  -d '{"path": "/home/PuMengYu/.config/clash/config.yaml"}'

# 方法二：先手动下载新订阅覆盖配置，再重载
# （订阅链接在机场网站/客户端里找）
wget -O /home/PuMengYu/.config/clash/config.yaml "你的订阅链接"
curl -s -X PUT http://127.0.0.1:9091/configs \
  -H "Content-Type: application/json" \
  -d '{"path": "/home/PuMengYu/.config/clash/config.yaml"}'

# 方法三：重启 Clash（会中断所有连接）
pkill clash
nohup /home/PuMengYu/clash -f /home/PuMengYu/.config/clash/config.yaml > /tmp/clash.log 2>&1 &
```

### 端口冲突问题

```bash
# 检查 7891 / 9091 端口是否被占用
ss -tlnp | grep -E '7891|9091'

# 若端口被占且 Clash 没在跑（残留进程），强制释放
pkill -9 clash
# 再重启 Clash
nohup /home/PuMengYu/clash -f /home/PuMengYu/.config/clash/config.yaml > /tmp/clash.log 2>&1 &
```

### 订阅节点更新后切回自动选择

更新订阅后节点名可能变化，务必重新切到自动选择让 Clash 重新测速：

```bash
curl -s -X PUT http://127.0.0.1:9091/proxies/%F0%9F%9A%80%20%E8%8A%82%E7%82%B9%E9%80%89%E6%8B%A9 \
  -H "Content-Type: application/json" \
  -d '{"name":"♻️ 自动选择"}'
```
